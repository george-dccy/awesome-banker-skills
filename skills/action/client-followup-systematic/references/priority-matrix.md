# 优先级矩阵

## 四象限判断框架

### 紧迫度 × 影响度

```
                    影响度高
                         │
          【第二象限】     │     【第一象限】
          重要但不紧迫     │     重要且紧迫
          ────────────────┼────────────────
          【第三象限】     │     【第四象限】
          不重要不紧迫     │     紧迫但不重要
                         │
                    影响度低
          ◀────────────────┼────────────────▶
                    紧迫度低         紧迫度高
```

### 第一象限：重要且紧迫（立即处理）

特征：

- 时间窗口即将关闭
- 对客户关系或业务有重大影响
- 对方正在等待我方动作

典型事项：

- 客户紧急询问需要当日回复
- 审批流程进入关键节点
- 合同签署窗口期
- 客户高层明确关注的进展

处理原则：立刻处理，不等不靠

### 第二象限：重要但不紧迫（计划处理）

特征：

- 对长期关系或业务有重大影响
- 目前没有紧急时间压力
- 需要系统规划和资源准备

典型事项：

- 客户年度合作方案规划
- 产品上线后的长期跟踪
- 客户关系深度经营计划

处理原则：纳入计划，设置检查点，保持节奏

### 第三象限：不重要不紧迫（尽量避免）

特征：

- 消耗资源但产出有限
- 对客户关系和业务无实质影响

典型事项：

- 例行但不产生价值的汇报
- 可以简化的行政流程
- 对方的低优先级请求

处理原则：简化、委托、或重新评估是否必要

### 第四象限：紧迫但不重要（尽快处理）

特征：

- 有时间压力但影响有限
- 通常是他人的优先事项转移到我方

典型事项：

- 对方催促的次要材料补充
- 临时的信息核实请求
- 配合第三方的时间敏感请求

处理原则：快速处理，不占用核心资源

## 优先级排序规则

1. **第一象限最优先**：重要且紧迫的事项立即处理
2. **第二象限保持节奏**：重要但不紧迫的事项按计划推进，不能因为不紧迫就忽视
3. **第四象限快速收尾**：紧迫但不重要的的事项尽快处理完，释放资源
4. **第三象限降级或放弃**：不重要不紧迫的事项降级处理

## 多事项并行时的处理

当同时有多个待跟进事项时：

1. 先排出优先级顺序
2. 识别可以并行处理的事项
3. 识别必须串行处理的事项（先后依赖）
4. 估算每个事项预计花费时间
5. 确定当日/本周能处理的事项数量
6. 对无法处理的事项设置明确的推迟理由和后续时间点

## 优先级重评触发条件

以下情况需要重新评估优先级：

- 客户高层介入或关注度变化
- 时间窗口发生重大变化
- 事项本身的意义发生变化
- 资源条件发生重大变化
- 超过一周没有进展的事项
