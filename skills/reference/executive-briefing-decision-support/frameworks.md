# 汇报判断框架

## 框架一：汇报类型判断矩阵

每个汇报在使用前必须先判断类型，类型决定结构。

### 三种汇报类型

| 类型 | 定义 | 领导角色 | 结构要求 |
|------|------|----------|----------|
| **同步型** | 让领导知道进展/结果 | 被告知 | 简洁摘要 + 关键事实 |
| **请求型** | 需要领导给资源/授权 | 决策者 | 结论先行 + 支撑逻辑 |
| **说服型** | 需要领导改变主意/批准方案 | 被说服 | SCQA 结构 + 风险对应对 |

### 判断规则

```
第一步：这件事不汇报会怎样？
  ├── 什么事都不发生 → 同步型（重新考虑是否值得汇报）
  └── 什么事会卡住/有风险 → 进入第二步

第二步：需要领导做什么？
  ├── 给意见/表态 → 同步型（加决策点）
  ├── 给资源/人/时间 → 请求型
  └── 改变原决定/批准方案 → 说服型
```

## 框架二：结构选择判断（结论先行 vs. 背景先行）

### 什么时候结论先行

来自 Pyramid Principle 的核心原则：

- 请求型汇报（要资源、要批准）
- 领导时间紧
- 已有明确建议动作
- 结论有冲击力

**结构**：结论 → 关键事实 → 当前卡点 → 建议动作 → 需要拍板项

### 什么时候背景先行

来自 SCQA 框架的改编：

- 说服型汇报（要改变领导主意）
- 领导对背景不熟悉
- 问题复杂需要铺垫
- 结论会出乎意料

**结构**：背景（S） → 冲突（C） → 问题（Q） → 答案（A）

### 银行场景改编

银行内部汇报有其特殊性：

| 场景 | 推荐结构 | 原因 |
|------|----------|------|
| 季度经营分析汇报 | 背景先行 | 领导需要先建立整体印象 |
| 问题升级要资源 | 结论先行 | 时间紧，领导要快速判断 |
| 方案审批汇报 | SCQA | 需要铺垫逻辑说服 |
| 风险预警汇报 | 结论先行+风险对应对 | 紧迫性优先 |
| 常规工作同步 | 结论先行+简略事实 | 效率优先 |

## 框架三：决策点识别四问

每个汇报前，必须自己回答这四个问题：

### Q1：这次汇报要让领导做几个决定？

- 1个：聚焦这一个，打透
- 2-3个：需要排序，主决策在前
- 3个以上：考虑拆成多次汇报

### Q2：每个决定的选项是否已经收敛到2-3个？

- 没有收敛 → 先内部讨论，汇报前先做选择
- 已收敛 → 确保每个选项都有支撑材料
- 领导可能自己提出新选项 → 提前准备否决理由

### Q3：每个决定的风险是否已经预判？

来自 BCG 决策框架的改编：

```
风险预判三问：
1. 如果领导同意，最坏结果是什么？（需要什么检查点）
2. 如果领导拒绝，有退路吗？（B方案是什么）
3. 如果领导质疑我的核心假设，我有没有备选数据？
```

### Q4：需要领导给的资源是否明确？

- 资源类型：时间、人力、预算、权限、信息
- 量化程度：模糊（"需要支持"）vs. 明确（"需要您拨2小时参加评审会"）
- 资源与决策捆绑：说明白给这个资源换什么回报/目标

## 框架四：领导质疑预判矩阵

来自麦肯锡七步法的"反对意见预演"步骤改编：

### 常见质疑类型

| 质疑类型 | 领导潜台词 | 提前准备 |
|----------|------------|----------|
| 数据质疑 | "这数据哪来的？" | 准备数据来源页 |
| 逻辑质疑 | "这个因果关系成立吗？" | 准备推导过程 |
| 优先级质疑 | "为什么这事排在前面？" | 准备对比说明 |
| 资源质疑 | "值不值得花这么多资源？" | 准备ROI或紧迫性说明 |
| 风险质疑 | "这事有什么风险？" | 准备风险对应对 |
| 权限质疑 | "这事不在你范围内吧？" | 准备授权说明 |

### 预判流程

```
第一步：列出领导可能质疑的3个最强点
  └── 不是所有点，而是最强的3个

第二步：每个质疑点准备30秒应答
  └── 不要长篇大论，30秒说不清的结构本身有问题

第三步：判断哪个质疑点可能让领导推翻结论
  └── 如果有，SCQA 结构
```

## 框架五：汇报结构选择流程

```
第一步：判断汇报类型
  ├── 同步型 → 结论先行，简洁事实
  ├── 请求型 → 结论先行，明确资源
  └── 说服型 → SCQA，铺垫逻辑

第二步：识别决策点数量
  ├── 1个 → 单线结构
  ├── 2-3个 → 主次分清
  └── 3个以上 → 考虑拆分

第三步：预判质疑点
  ├── 有强质疑点 → SCQA 或加风险对应对
  └── 无强质疑点 → 结论先行

第四步：确认资源需求
  └── 明确量化资源与交换条件
```
