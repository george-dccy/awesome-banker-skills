# 豆包入口说明

## 为什么单独做豆包入口

这个仓库重点适配豆包的使用方式。  
目标不是给出一个很长的万能 prompt，而是提供几张“可直接复制、可多轮持续使用、角色清楚”的入口卡片。

这类 prompt 的关键不是把“怎么读仓库”完全藏起来，而是要同时满足两点：

- 对模型内部要求足够明确：先读 registry，再读对应正文，默认不要绕开仓库去外搜
- 对最终回答保持足够自然：用户看到的是专业结论、判断和建议，而不是一串技术过程
- 首轮和后续追问都继续以仓库为准，而不是只第一轮用仓库
- 复制时就能直接提问，所以 prompt 末尾要保留“我的问题”占位

## 入口列表

### 1. 公开咨询入口

- 文件：`prompts/entrypoints/doubao/public-consulting.md`
- 适合：客户、公开咨询用户、非专业用户
- 特点：先用公开知识包回答，支持围绕同一产品连续追问

### 2. 银行员工入口

- 文件：`prompts/entrypoints/doubao/bank-staff.md`
- 适合：客户经理、产品协同、业务支持人员
- 特点：强调岗位判断、推进顺序、话术建议，并支持同一事项的多轮推进

### 3. 基层管理者入口

- 文件：`prompts/entrypoints/doubao/frontline-manager.md`
- 适合：团队负责人、支行长、基层管理者
- 特点：强调判断、拆任务、盯进度、做闭环，并支持围绕同一管理事项持续对话

### 4. 总行领导层入口

- 文件：`prompts/entrypoints/doubao/head-office-leadership.md`
- 适合：总行或分行中高层
- 特点：强调核心判断、关键事实、风险取舍和拍板建议，并支持同一决策事项的多轮追问

### 5. 自动路由入口

- 文件：`prompts/entrypoints/doubao/auto.md`
- 适合：不确定该用哪个入口时
- 特点：自动识别身份和场景，适合作为多轮对话的通用入口

## 统一要求

所有豆包入口都要求模型：

- 先读取 `registry/*.json` 或 `registry/knowledge-packs.json`
- 先按问题类型去读对应正文，不要只停留在入口 prompt 本身
- 遇到产品或公开知识问题，优先依据对应 `knowledge-packs/*`
- 默认不要用外部搜索替代仓库内容
- 这套仓库约束适用于整个对话，不是只第一轮生效
- 先识别身份、岗位和问题场景
- 先直接给结论、判断或下一步动作
- 需要时再补“工作判断依据”和“公开资料依据”
- 少讲技术术语，少展示内部路由过程
- prompt 末尾保留“我的问题：{{...}}”占位，方便直接复制使用
- 覆盖不足时明确写出“当前仓库未覆盖”

## 使用建议

- 如果你已经知道自己的身份，优先用对应入口
- 如果你只想先试一试，用 `auto.md`
- 如果你需要把 private overlay 也算进去，建议在本地 Agent 环境中再叠加 `workspace/private/registry/*`
