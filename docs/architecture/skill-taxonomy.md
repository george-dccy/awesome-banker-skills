# Skill 分类说明

## 为什么不再保留 role / workflow / method 三分法

因为这三者不是同一维度：

- `role` 更像身份或视角
- `workflow` 更像任务编排
- `method` 更像通用框架

继续并列下去，结构会越来越难解释，也越来越难触发。

新的做法是：

- 顶层只保留两类 skill：`reference` 和 `action`
- 真正的分类学放进 `capability + trigger metadata`

## 两类 skill

### `skills/reference`

回答的问题是：

- 应该怎么看
- 应该怎么判断
- 应该怎么表达

适合承载：

- 专业视角
- 判断框架
- 表达结构
- 优先级规则
- 常见误区

### `skills/action`

回答的问题是：

- 这个任务该怎么做
- 需要什么输入
- 按什么顺序推进
- 最终要产出什么

适合承载：

- 首访推进
- 会后跟进
- 汇报拍板
- 蒸馏整理

## capability 标签

当前统一使用五类 capability：

- `understand`
- `analyze`
- `execute`
- `communicate`
- `distill`

它们分别对应：

- `understand`：理解概念、产品、机构、机制
- `analyze`：拆问题、抓变量、做判断、做对比
- `execute`：推进任务、安排动作、形成闭环
- `communicate`：表达、汇报、写作、沟通
- `distill`：把资料、经验、偏好沉淀成长期资产

## 为什么 knowledge 不并进 skill

因为事实性知识和程序性 skill 的更新逻辑不同。

- `knowledge` 负责公开事实、FAQ、来源
- `skills` 负责如何看、如何做、如何表达

这能让仓库既有专业方法，也有公开依据，但不会混层。
